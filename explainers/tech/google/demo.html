Africa-south1. Two services live.

The actual commands, outputs, and architectural takeaways from standing up the first 2nth.ai GCP project. Compute Engine and Cloud Run validated in Johannesburg (africa-south1) — and the single org-policy gotcha that shaped the architectural pattern for everything coming next.

00 · Setup

Switch identity and project.

The craig@2nth.ai Workspace identity was freshly created; gcloud was still bound to the previous craig@agilex.co.za account. One revoke, one login, one project switch — and the CLI is pointing at the right surface.

terminal · gcloud auth + project switch
# revoke the old account
gcloud auth revoke craig@agilex.co.za

# authenticate as the new identity (browser OAuth)
gcloud auth login craig@2nth.ai --update-adc
Application Default Credentials (ADC) were updated.
You are now logged in as [craig@2nth.ai].

# point at the POC sandbox project
gcloud config set project cs-poc-kzmkho6ivq8mhceekpqttth

# fix the ADC quota project mismatch warning
gcloud auth application-default set-quota-project cs-poc-kzmkho6ivq8mhceekpqttth

The POC project is a Google-auto-generated sandbox (the cs-poc- prefix is the giveaway) sitting inside folder 436629319555. It comes with inherited org policies — a fact that will matter in a minute.

01 · Compute Engine · ✓ passed

A private VM. IAP-only SSH. Three minutes in, three minutes out.

The question being tested: can we stand up a VM in africa-south1, reach it without a public IP, and have everything torn down before it costs anything? Yes to all three.

terminal · enable APIs + provision VM
# enable the three APIs we need
gcloud services enable compute.googleapis.com iap.googleapis.com oslogin.googleapis.com

# firewall rule: allow IAP's range to reach tag iap-ssh on tcp:22
gcloud compute firewall-rules create allow-iap-ssh \
  --network=default --direction=INGRESS --action=ALLOW \
  --source-ranges=35.235.240.0/20 \
  --rules=tcp:22 \
  --target-tags=iap-ssh

# create e2-micro VM, NO external IP
gcloud compute instances create test-gce-01 \
  --zone=africa-south1-a \
  --machine-type=e2-micro \
  --image-family=debian-12 --image-project=debian-cloud \
  --boot-disk-size=10GB --boot-disk-type=pd-balanced \
  --no-address \
  --tags=iap-ssh \
  --scopes=cloud-platform \
  --metadata=enable-oslogin=TRUE

What this proved

  • af-south1 provisioning path works for general compute. 3 zones (a/b/c), all UP.
  • Private-only networking is viable — no external IP, no exposed SSH port, no bastion VM.
  • IAP SSH tunneling works for an owner identity (after a quick iap.tunnelResourceAccessor grant) — the path teammates will use.
  • Metadata server + default Compute SA are reachable from inside the VM — the foundation for Workload Identity and SDK auth without JSON keys.
  • Debian 12 image is current (built 3 days before the test run) — no stale base for new VMs.

02 · Cloud Run · ✓ passed (with a twist)

Serverless container. Scale to zero. Auth-only by design.

The 2nth default for GCP compute. Pay-per-request, scale to zero, global HTTPS out of the box. Deploy Google's own hello container — no Dockerfile, no build, just the image.

terminal · enable APIs + deploy
gcloud services enable \
  run.googleapis.com \
  artifactregistry.googleapis.com \
  cloudbuild.googleapis.com

gcloud run deploy hello \
  --image=us-docker.pkg.dev/cloudrun/container/hello \
  --region=africa-south1 \
  --allow-unauthenticated \
  --min-instances=0 --max-instances=2 \
  --memory=256Mi --cpu=1

What this proved

  • Cloud Run in af-south1 deploys cleanly — revision created, traffic routed, service URL live in under 90 seconds.
  • Scale-to-zero works — min-instances=0 means no instances running after idle timeout, no billing between requests.
  • Identity-token auth works — the same pattern a Cloudflare Worker will use when calling this Cloud Run service via a signed Google ID token (see the compute skill for the Web Crypto JWT signer).
  • Deploy-by-image works without a build step — Google's public Artifact Registry serves a cloudrun/container/hello image that bypasses Cloud Build entirely.
03 · The gotcha that shapes the architecture

iam.allowedPolicyMemberDomains — the inherited policy that changes everything.

Both tests surfaced the same constraint. When adding an IAM binding for craig@2nth.ai or for allUsers, the call is rejected:

terminal · the rejection
description: User craig@2nth.ai is not in permitted organization.
subject: orgpolicy:projects/cs-poc-kzmkho6ivq8mhceekpqttth
type: constraints/iam.allowedPolicyMemberDomains

The mechanism

An allowed-domains policy is inherited from the org folder, and 2nth.ai isn't on the list.

The POC project sits inside folder 436629319555. That folder has an allowed-domains org policy that restricts which Workspace domains can hold IAM roles. It doesn't include 2nth.ai — so neither adding a 2nth.ai user nor the public allUsers principal succeeds.

Neither test was blocked in practice: Craig already had owner rights via the project's initial grant, and the Cloud Run service deployed fine, it just defaulted to authenticated-only. But the policy is going to shape every future design decision.

The architectural consequence

Public-facing 2nth.ai products can't live directly on this GCP project — no allUsers means no open-to-the-world Cloud Run service, no public bucket, no anonymous function endpoint.

That forces the architecture we already wanted anyway. Cloudflare Workers hold the public edge — TLS, rate limiting, auth, caching, WAF. They sign a Google ID token from a service-account key stored as a Worker secret, and call the private Cloud Run service over HTTPS. Every public feature becomes: edge Worker → private Cloud Run → private data plane. The org policy is effectively enforcing the hybrid pattern.

For teammates (adding james@2nth.ai, katharine@2nth.ai), either (a) ask the org admin to add 2nth.ai to the allowed-domains list on folder 436629319555, or (b) move the project to a folder with a broader policy.
