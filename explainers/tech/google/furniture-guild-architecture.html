Architecture & review · built 19 April 2026

Furniture Guild — stack, wiring, review.

A checkpoint. Over one day we stood up a customer-facing storefront, a back-office production dashboard, a Cloudflare Worker API gateway, a Cloudflare Tunnel into a private GCE VM, and a fully-seeded ERPNext with 16 products, 12 customers, opening stock, invoices, and work-in-progress BOMs — all POPIA-resident in africa-south1. This page is for reading and review before we push further.

Live · 3 Pages projects Live · 1 Cloudflare Worker Live · 1 Tunnel (Johannesburg) Live · ERPNext v16 + MariaDB 2 open blockers
01 · The goal

Purpose-built front-ends on a shared ERPNext.

ERPNext is operationally brilliant and visually brutal. Most SA businesses that need a proper system of record (sales, stock, accounting, manufacturing) end up living with its default UI, paying for a SaaS that abstracts it, or building partial tools in spreadsheets. The 2nth pattern is a third path: keep ERPNext as the system of record, wrap it in purpose-built views — one for customers, one for the workshop floor, one for finance — all backed by the same data.

Furniture Guild is the first instance. A fictional SA furniture manufacturer (solid oak / yellowwood / walnut / pine, Cape Town workshop, selling to retail chains and direct) was chosen as a recognisable, non-trivial business shape — manufacturing, inventory, trade accounts, VAT, multi-step orders.

Properties of this shape: the VM has no public IP, no open ports, no inbound connectivity that wasn't initiated from inside. The Tunnel is the only ingress, and it authenticates to Cloudflare with an account-scoped cert. The Worker is the only thing with the ERPNext API key, so the browser never sees it. Inputs are validated at the Worker boundary, never trusted at the storefront.

04 · Design decisions

Nine calls made along the way.

Each with the alternative, and the reason we picked what we picked. Flagging these so any you disagree with can be reversed now, before the pattern calcifies.

01 · Compute shape
ERPNext on a GCE VM, not on Cloud Run

Picked: Compute Engine e2-standard-2 with Docker Compose.

Alt: Cloud Run + Cloud SQL + Memorystore. Why not: Frappe's filesystem assumptions (logs, site files, `assets/`) don't play nicely with read-only container filesystems without extra mount gymnastics. VM is simpler for POC, cheaper for one-tenant, revisit when adding customers or scaling.

02 · VM networking
Private-only VM with Cloud NAT, not a public IP

Picked: --no-address + Cloud NAT for outbound, IAP for inbound SSH, Cloudflare Tunnel for inbound HTTP.

Alt: VM with external IP + firewall rules. Why not: public IPs are a standing attack surface; inherited org policy iam.allowedPolicyMemberDomains already prevents allUsers on Cloud Run, so public-facing is off the table anyway. The Tunnel pattern generalises to everything we'll add later.

03 · Public access path
Cloudflare Tunnel, not public LB + VPN

Picked: cloudflared running as a systemd service on the VM, tunnelling to erp-fg.2nth.ai.

Alt: GCP HTTPS Load Balancer + IAP-protected backend. Why not: Tunnel is free, no external IP needed, TLS + DDoS at Cloudflare edge included, matches the rest of the 2nth stack. The LB path adds ~$20/mo and doesn't improve security here.

04 · API gateway layer
Cloudflare Worker fronts ERPNext — browser never sees the key

Picked: furnitureguild-api Worker with the ERP API token as a secret; storefront POSTs to the Worker; Worker signs the request into ERPNext.

Alt: Browser calls ERPNext directly, API key in a hidden (but reachable) config. Why not: any secret in the browser is stolen within minutes of the page going live. Worker also gives us input validation, CORS gating, rate-limiting, and audit-log opportunities on a single chokepoint.

05 · ERPNext authentication
Dedicated narrow-scope API user, not Administrator

Picked: storefront-api@furnitureguild.local with Sales User + Sales Manager roles.

Alt: Use the Administrator token. Why not: separates the audit trail (documents show the right author), minimises blast radius if the Worker secret leaks, and makes a scoped role profile easy to swap later if we want to remove Sales Manager.

06 · Hostname / zone
erp-fg.2nth.ai — on an existing zone, not a new domain

Picked: subdomain of the existing 2nth.ai Cloudflare zone.

Alt: buy furnitureguild.co.za. Why not: this is still a POC / internal demo. If Furniture Guild becomes a real engagement we buy the real domain and swap the CNAME. Zero rework.

07 · Consumer UI — distinct design language
Storefront uses a warm wood-palette, not the know-2nth technical theme

Picked: Furniture Guild has its own visual identity (wood/cream/sage) distinct from the know-2nth blue/sky.

Alt: Reuse the technical theme. Why not: clients need to see that the 2nth stack makes anything visually possible — not that every app looks like know-2nth. The workshop dashboard, by contrast, adopts the dense technical theme because it's an internal tool.

08 · Data seeding
Real catalogue + real opening stock + real invoices, not mocked

Picked: 16 furniture items with ZAR pricing, 12 SA retail customers, opening stock valued R 976k in Finished Goods + R 164k in Raw Materials, historical sales invoices submitted to create real GL entries.

Alt: Mock data in localStorage on the storefront. Why not: the whole demo is about showing that ERPNext + 2nth wrapping works. Mock data removes the ERPNext from the demo.

09 · Two separate storefront + workshop repos vs one
Keep them together in furnitureguild — for now

Picked: both HTML pages in the 2nth-ai/furnitureguild repo, same Pages project.

Alt: separate furnitureguild-workshop repo + Pages project. Why not: they share context, data, and design language. If the workshop view grows to need backend state or per-user auth, it moves to its own deployment — but until then, one repo + simple branches is faster.

05 · Known issues

What's not done, and what's wrong.

Worker subdomain DNS not resolving yet

Deployed at furnitureguild-api.<subdomain>.workers.dev, but DNS is stale / the workers.dev subdomain for this account may need re-verifying. Impact: blocks the storefront → Worker wiring test. Workaround: test the Worker locally via wrangler dev; custom domain (e.g. fg-api.2nth.ai) also unblocks this.

Storefront still uses the simulated quote flow

The "Request quote" button on furnitureguild.pages.dev runs local JS and shows a fake quote number. Real wiring is ~10 lines of fetch()-to-Worker JS — waiting on the Worker DNS above.

Workshop dashboard data is baked, not live

The progress bars, stock levels, orders and alerts on /workshop.html are static. Hooking them to GET /quotes on the Worker (and a couple of new endpoints for stock + work orders) is the next round after the storefront wiring.

Manufacturing BOMs half-landed

The BOM seed script hit two API issues (Operation doctype expected name not operation_name; BOMs blocked on the missing operations). Raw materials + opening stock landed cleanly; BOMs + Work Orders still as drafts. Workshop dashboard shows plausible work-order numbers but they're not linked to real BOMs.

ERPNext running on defaults

Administrator password is admin, MariaDB root password is admin, Cloudflare Access is not in front of erp-fg.2nth.ai. Fine for a demo where only the Worker (with its own token) should hit ERPNext. Not fine if anyone needs to log in directly — at which point we rotate passwords and add CF Access.

No payment webhook yet

The storefront shows "Accept & pay 50% deposit" but no payment provider (PayFast / Ozow / Stripe) is wired. A real payment flow needs a separate Worker route handling the webhook callback, creating a Payment Entry in ERPNext.

ERPNext host_name not set

Emails and PDF invoices generated by ERPNext will include URLs pointing to localhost:8080 / container names, not erp-fg.2nth.ai. One-command fix (bench set-config host_name ...), cosmetic until we enable outbound email or PDFs.

06 · For review

Questions I need answers to before we keep building.

Architecture direction

Do we keep ERPNext on a single GCE VM, or is Cloud Run + Cloud SQL the target shape? GCE is simpler for one tenant. Cloud Run + Cloud SQL is the right multi-tenant answer and lets us offer Furniture Guild as an installable template to real clients.
Is erp-fg.2nth.ai fine long-term, or do we want a neutral domain for the demo (e.g. erp.furnitureguild.co.za if we buy the domain)? Subdomain of 2nth.ai is fine for now. If Furniture Guild shows up on a customer site, we need a white-labelled domain.
Should the storefront stay on Cloudflare Pages, or move to something that handles server-side rendering (Workers / Astro / Next) when we add customer accounts? Pure static Pages hit a wall the moment we need per-user state. Migration is cheap if we do it before too much styling locks in.

Security posture

Cloudflare Access in front of erp-fg.2nth.ai now, or when someone actually needs to log into ERPNext through the UI? Right now the URL serves the Frappe login page publicly. CF Access would gate that page entirely (require a Google login to even see it).
Rotate admin / MariaDB passwords now, or batch with the CF Access hardening? Password rotation is ~5 min of work. If we set up a real operator user, Administrator can be suspended entirely.
Should the Worker be on a custom domain (fg-api.2nth.ai) instead of *.workers.dev? Custom domain is free, makes the domain story coherent, and adds Cloudflare DNS security policies. Answers the current DNS blocker too.

Scope direction

Next priority: finish storefront → Worker wiring (quote submission) or start the workshop live-data wiring (read-side)? Submission is the "wow" moment for a demo; live workshop data makes ERPNext feel responsive. Both are ~half a day each.
Build a second demo company after Furniture Guild, or polish this one further (manufacturing, payments, customer accounts)? Second company shows the pattern generalises (e.g. a B2B services firm, a restaurant group). Polishing shows depth. Probably polish first, widen once real engagements need it.
Is this the right moment to open up an API key + read-side view for a real partner to try? If any existing 2nth partner has a use case where Furniture-Guild-as-template fits, now is the moment to invite them. Feedback beats guessing.
07 · If we just keep building

Shortest paths from here.

  1. ~30 min — Worker custom domain fg-api.2nth.ai (fixes DNS blocker); set host_name in ERPNext
  2. ~2 hours — Storefront → Worker wiring. Real SAL-QTN-* numbers on submission. Small demo script showing the QTN appearing in ERPNext UI.
  3. ~3 hours — Workshop live-data: GET /quotes, GET /orders, GET /stock on the Worker; storefront/workshop auto-refresh every 30s.
  4. ~1 day — Payment webhook (pick one provider), Payment Entry flow, customer-side order tracker that reads real status.
  5. ~2 days — Customer accounts via Cloudflare Access email-OTP, customer-side tracker shows only their own orders.
  6. ~1 week — Manufacturing BOMs fixed, Work Order flow end-to-end, mobile workshop-tablet view for the production floor.
Where this is now

One day from first signup to end-to-end pipeline.

Everything listed above was built between first-signup to craig@2nth.ai's GCP account and this writing — approximately one working day. Two architectural surprises shaped the final design: the inherited iam.allowedPolicyMemberDomains org policy (which forced the "Cloudflare edge fronts private compute" pattern earlier than planned), and the pwd.yml encryption_key gap (which cost 20 minutes to debug once and is now saved so it doesn't cost anyone 20 minutes again). Neither was a blocker; both sharpened the architecture.
