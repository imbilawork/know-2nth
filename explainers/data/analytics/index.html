01 · Tools

The analytics toolkit.

Six tools that cover different points on the BI spectrum — from full-featured SQL exploration to enterprise-grade dashboarding to code-first reporting to real-time OLAP. Superset and Tableau are live; the rest land as content ships.

Apache Superset
Live
data/analytics/superset

Open-source BI platform. SQL Lab for ad-hoc queries, 40+ chart types, dashboard authoring, embedded analytics via guest tokens. The 2nth default — preferred over Power BI and Tableau for new builds.
Tableau
Live
data/analytics/tableau

Salesforce's BI platform. Tableau Cloud / Server / Desktop, plus Tableau Pulse (AI-monitored metrics) and Tableau Next (agentic analytics inside AgentForce). Zero-copy integration with Salesforce Data Cloud. The path of least resistance if you're already on Salesforce.
Metabase
Stub
data/analytics/metabase

Self-service BI with a friendlier learning curve. Point-and-click questions, embeddable dashboards, and a community edition that covers most small-to-mid reporting needs without writing SQL.

Evidence
Stub
data/analytics/evidence

Code-first BI with markdown reports. Write SQL in .md files, generate interactive pages, version control your dashboards alongside your dbt models. For teams that live in the IDE.

Apache Druid
Stub
data/analytics/druid

Real-time OLAP for sub-second queries on event streams. Not a BI tool in the traditional sense — more the engine underneath one. Pairs with Superset as the query backend for high-cardinality, time-series-heavy workloads.

Grafana
Stub
data/analytics/grafana

Metrics, logs, traces visualization. Primarily an ops tool, not a BI tool — but the line blurs when your infrastructure telemetry is the dataset. Loki, Tempo, Mimir, and the observability stack.

02 · The shared pattern

Every BI tool sits on the same architecture.

The tools in this sub-hub differ in UI and target audience, but they all follow the same four-layer architecture. Understanding this pattern makes every individual tool easier to reason about.

Warehouse → Semantic layer → Dashboard → Consumer

Every modern BI deployment follows the same flow. Data lands in a warehouse or lakehouse (PostgreSQL, ClickHouse, DuckDB, BigQuery). A semantic layer defines the business logic — metrics, dimensions, relationships — so that every dashboard computes the same number for "revenue" instead of each analyst inventing their own SQL. The dashboard tool queries the semantic layer and renders charts. The output is consumed via browser, embedded iframe, scheduled email, or API.

# The four-layer BI architecture

Source systems  →  ERP, CRM, logs, events, APIs
      ↓
Warehouse       →  PostgreSQL / ClickHouse / DuckDB / BigQuery
      ↓               (dbt transforms raw → staging → marts)
Semantic layer  →  Superset datasets / Metabase models / cube.dev
      ↓               (metrics, dimensions, row-level security)
Dashboard tool  →  Superset / Metabase / Evidence / Grafana
      ↓
Consumers      →  Browser / embedded iframe / email / Slack / API

The payoff: swap any layer without rebuilding the stack. Move from PostgreSQL to ClickHouse without touching your dashboards. Switch from Metabase to Superset without re-modeling your data. The architecture is the constant; the tools are the variables.
