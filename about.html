About · access & member management

Three tiers, one sign-on, no shadow paywall.

Access on know.2nth.ai is honest by design. Open content stays open — no email gates, no soft paywalls, no friction. Member content is new content classes (operational nodes, deployment templates, role-based bundles) authored to compound for teams that ship. Partner is bespoke. Sign-in is shared across the whole 2nth.ai estate.

Live Open / Member / Partner SSO via beta.2nth.ai Cookie scope: .2nth.ai v1.0
01 · Philosophy

Open stays open.

The access model is built on a simple promise: technology explainers are always free. Every leaf in the existing tree — framework deep-dives, decision guides, the agent-context format itself — is readable without an account. That promise is on the access page; it's load-bearing.

Member is not a paywall on existing depth. It's a separate content class. Operational skill nodes, deployment templates, role-based context bundles, curated learning paths — the content classes that take work-shaped form and need community context. This kind of content does not exist by accident; it is authored deliberately, and it is what Member is for.

Partner is bespoke: private branches of the tree for organisations whose IP becomes part of the ecosystem. Co-branded domain nodes, agent-context packages keyed to one company's stack, institutional knowledge capture as an ongoing service. Different commercial frame, different intake flow.

What we deliberately don't do

We don't email-gate explainers. We don't dim the second half of a page. We don't run popups asking for newsletter signups. The Open / Member / Partner architecture lets the access page tell the truth: everything labelled "All technology explainers" actually is. If you're seeing depth, you can keep reading.

Why two backend names map to "Member"

The estate's billing layer (on beta.2nth.ai) supports four plans: explorer, starter, designer, builder. skills.2nth.ai exposes them under those names because skills consumption can fan out across four price points. know.2nth.ai consolidates the middle two into one display tier, "Member", because the access model on the knowledge tree is binary at the threshold — you either see the operational depth or you don't. The auth bar on this site shows "Member" for either backend plan.

03 · Authentication

Single sign-on across the .2nth.ai estate.

Authentication is centralised on beta.2nth.ai. Cookies are set with Domain=.2nth.ai, which means a session is recognised on every subdomain — sign in once, all surfaces see you. The auth bar on this site, on skills.2nth.ai, and inside the beta.2nth.ai dashboard all read the same session payload.

The mechanics, in order:

01
Page loads, know-auth.js runs at the bottom of the body. It calls fetch('https://beta.2nth.ai/api/auth/session', { credentials: 'include' }) with the .2nth.ai cookie attached.
02
The auth API responds with { user } — either a populated object (signed in) or null (signed out). The script stashes it on window.__2nth_user.
03
Auth bar renders. Signed out: a "Sign In" pill linking to beta.2nth.ai/join.html. Signed in: a tier badge (Open / Member / Partner) plus an avatar with initials; clicking the avatar opens a dropdown with Dashboard, Projects, Upgrade, Sign out.
04
applyTierGates() walks the DOM looking for data-tier="open|member|partner" attributes. When MEMBER content ships, those attributes will trigger an upgrade overlay for visitors below the required tier. Currently no element on the site carries the marker — explainers stay open.
05
window.__2nth_canAccess(tier) is exposed for inline use, returning true / false based on the current session. Useful for inline <script> blocks that want to render different copy based on the visitor's plan.

Cookie scope, in one sentence

The session cookie is set by beta.2nth.ai with Domain=.2nth.ai, HttpOnly, Secure, SameSite=Lax — so the browser sends it to every *.2nth.ai surface but to no one else. That single cookie is the entire SSO mechanism.

04 · Member signup

From "Sign up" to active session.

There's only one signup endpoint: beta.2nth.ai/join.html. It runs the entire estate. From know.2nth.ai/#access, the "Sign up for Member access" button drops the visitor straight into it.

01
Visitor clicks Sign up for Member access on know.2nth.ai/#access → redirected to beta.2nth.ai/join.html.
02
They register with email + password. Default plan: explorer (free).
03
Session cookie set with Domain=.2nth.ai. The visitor is now signed in across all 2nth.ai surfaces.
04
If they want Member access, they upgrade to starter or designer at beta.2nth.ai/bill.html. The auth bar updates to show the Member badge on next page load.
05
Member content (when it ships, marked with data-tier="member") becomes visible. The upgrade overlay know-auth.js would otherwise show is replaced by the actual content.

What the auth bar tells you

If you see a tier badge in the top-right of the page, you're signed in. The badge colour distinguishes the three tiers: sky for Open, blue for Member, amber for Partner. The avatar dropdown holds the upgrade and sign-out controls.

05 · Partner intake

When private branches make sense.

The Partner tier is for organisations whose own IP — methodologies, codebase architecture, internal frameworks — should live as a co-branded branch of the 2nth.ai tree. It's a longer commercial conversation; intake is a real conversation, not a self-serve flow.

Partner intake starts at partners@2nth.ai. The "Request Partner intake" button on know.2nth.ai/#access opens a mailto: link with a pre-filled subject. The conversation typically covers:

  • Domain shape — what discipline does your IP belong to? It might map cleanly into an existing top-level domain (Business / Data / Design) as a sub-branch, or it might be its own.
  • Authoring cadence — private branches are kept fresh by either your team or ours; the choice affects the commercial terms.
  • Agent context packaging — how the resulting content is exposed to AI agents that consume it (alongside or instead of the public tree).
  • Institutional knowledge capture — many Partner engagements include translating tacit org knowledge (currently in heads or wikis) into the canonical-skill-node format.

Auth-wise, Partner is just a plan

Backend plan builder grants Partner-tier access on the auth side. The bespoke commercial relationship is layered on top — private subdomains, dedicated review cycles, separate publishing pipelines — but for the visitor experience, "Partner" is just one more rung of the same SSO ladder.

06 · Tier-gating mechanics

How content gets gated when MEMBER ships.

The hook is in place; no markers exist yet. When the first piece of MEMBER content lands, all that's needed is a single attribute on the appropriate element. know-auth.js handles the rest at page load.

To gate any element to a tier, add data-tier="open|member|partner":

<!-- Visible to everyone (no gate) -->
<section>...</section>

<!-- Visible to Member+ (signed in, starter or above) -->
<section data-tier="member">...</section>

<!-- Visible only to Partner (builder plan) -->
<section data-tier="partner">...</section>

<!-- Inline tier check inside a custom script -->
<script>
  if (window.__2nth_canAccess('member')) {
    // render member-only widget
  }
</script>

What happens at runtime when a visitor below the required tier hits a gated element:

  1. know-auth.js finds the element via document.querySelectorAll('[data-tier]')
  2. Checks the visitor's plan against the threshold mapping (open = 0, member = 1, partner = 3)
  3. If under threshold, wraps the element in a relative container and overlays an upgrade prompt with a CTA to beta.2nth.ai/bill.html
  4. Visitors at or above the threshold see the element normally; the gating code never runs for them

Authoring discipline

Two rules. (1) Never put data-tier="member" on existing explainer content — it breaks the access-page promise. (2) When MEMBER content (operational nodes, templates) lands, gate at the section level, not the page level — the visitor still gets a useful free preview before the gate, and the upgrade decision happens with context.

07 · Across the estate

One sign-in, every surface.

A session signed in once is recognised across every *.2nth.ai surface. Each surface presents the auth bar in its own visual language but reads the same session payload from the same cookie. The backend plan name is canonical; each surface translates it locally for display.

Logout, everywhere

window.__2nth_logout() POSTs to beta.2nth.ai/api/auth/logout, which clears the .2nth.ai-scoped cookie. The browser then has no session for any surface. The user is redirected to beta.2nth.ai/join.html; refreshing any other 2nth.ai tab shows the signed-out state.
